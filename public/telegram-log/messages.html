Changelog v0.4.3

ductor v0.4.3

Native Windows support, Telegram forum topic threading, and a cross-platform /stop fix.

Windows compatibility

ductor now runs natively on Windows without WSL. Key platform adaptations:

• CLI stdin mode: On Windows, prompts are passed via stdin instead of command-line arguments. The .CMD wrappers that npm generates for claude and codex mangle special characters in arguments — stdin bypasses this entirely.
• Process tree termination: /stop now uses taskkill /F /T on Windows to kill the entire process tree (cmd.exe + child node.exe), not just the direct subprocess.
• PID lock: os.kill(pid, 0) raises unexpected OSError variants on Windows. These are now caught so stale PID files no longer prevent startup.
• signal.SIGKILL: Does not exist on Windows. SIGKILL detection in the recovery path uses a safe fallback (getattr(signal, "SIGKILL", 9)).
• Timezone detection: Cron scheduling reads the system timezone via datetime on Windows instead of /etc/localtime. Jobs run at the correct local time out of the box.
• Clean Ctrl+C: KeyboardInterrupt from asyncio.run() is caught at top level — no more ugly tracebacks when stopping the bot.

Shared helpers (_IS_WINDOWS, _win_stdin_pipe(), _win_feed_stdin()) live in cli/base.py and are used by both Claude and Codex providers.

Telegram forum topic support

The bot now works correctly in Telegram groups with forum topics enabled:

• message_thread_id is extracted and propagated through all handler, sender, and streaming paths.
• Replies, streaming edits, typing indicators, and file sends all land in the correct topic thread.
• Sessions remain keyed by chat_id (no per-topic isolation).
• Background systems (cron, heartbeat, webhook) target private chats and are unaffected.

/stop now actually stops — immediately

Previously, /stop killed the CLI subprocess but the streaming loop kept reading buffered stdout until the process exited (up to 10+ seconds on Windows). Now:

• The streaming loop checks was_aborted() on each event and breaks out immediately.
• On Windows, the full process tree is terminated (not just the wrapper).
• This fix is cross-platform — Linux benefits equally.

Backward compatible

No config changes required. Existing setups benefit automatically.

Changelog v0.4.4

ductor v0.4.4

Native background service on all three platforms, provider-isolated sessions that survive CLI switches, and invisible CLI windows on Windows.

Cross-platform background service

ductor service install now works on macOS, Windows, and Linux — each with the platform-native daemon mechanism:

• macOS: Installs a launchd Launch Agent (~/Library/LaunchAgents/dev.ductor.plist). Starts on login, restarts only on crash (KeepAlive: {SuccessfulExit: false}), 10-second throttle. PATH includes Homebrew and NVM paths automatically.
• Windows: Creates a Task Scheduler logon trigger with pythonw.exe — no visible console window. Detects missing admin rights and prints clear guidance (CMD > right-click > Run as administrator). Restart delay reduced from 30s to 10s.
• Linux: systemd user service (extracted from the previous monolithic service.py into its own module). Unchanged behavior.

The platform dispatcher (service.py) auto-selects the correct backend based on sys.platform.

Provider-isolated sessions

Switching between Claude Code and Codex no longer discards the other provider's session:

• Each provider tracks its own session_id, message_count, total_cost_usd, and total_tokens.
• /model codex switches the active provider — switching back to Claude later resumes exactly where you left off.
• /status shows metrics for the currently active provider.
• /new resets only the currently active provider session. Other providers stay intact.
• Old single-session JSON files are migrated transparently on first load.

CLI errors no longer auto-reset sessions

Previously, a CLI error (token limit, timeout, network issue) would automatically discard the session and retry with a blank slate. Now:

• The session is preserved on error.
• The user sees: "Your session has been preserved — send another message to retry. Use /new to start a fresh session if the problem persists."
• SIGKILL auto-recovery still works but only resets the affected provider, not all of them.

Invisible CLI windows on Windows

When ductor runs as a background service on Windows, CLI subprocesses (Claude, Codex, Codex model discovery) no longer flash visible console windows:

• CREATE_NO_WINDOW flag applied to all asyncio.create_subprocess_exec calls.
• pythonw.exe stderr=None guard in logging config prevents crashes.
• ductor service logs now correctly finds agent.log (previously looked for wrong filename pattern).

Fixes

• Command routing: ductor service uninstall no longer non-deterministically triggers the bot uninstaller. Root cause: Python set iteration order is undefined — switched to list to preserve argument order.
• Init wizard: macOS-specific service description text.
• Model selector: reset_provider_session() replaces full reset_session() on model switch — only the target provider is cleared.

Backward compatible

No config changes required. Existing sessions.json files are migrated automatically. The ductor service subcommands use the same interface across all platforms.
